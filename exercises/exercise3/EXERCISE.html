Debugging Exercise - Inconsistent Item Pagination Metadata

Investigate why the list endpoint reports changing totals as pagination advances.

## 📋 Statement

QA flagged an issue with the items list API while validating pagination in the dashboard. The item rows themselves look correct, and moving to later pages returns the expected subset of records, but the count field in the response starts dropping once a non-zero skip value is used.

This only seems to happen on the paginated list endpoint when requesting later slices of the same dataset. Default usage without a skip parameter still looks fine, and single-item endpoints are unaffected. The frontend team noticed that pagination controls become inconsistent because the reported total appears to shrink as the user advances through the list.

The defect compounds in a predictable way: every time the request offset increases, the returned total decreases by the same amount, even when the underlying matching records have not changed.

## 🔴 Observed defective output

1. First page

GET /api/v1/items/?skip=0&limit=2

200 OK
{
  "data": [
    { "id": "a1", "title": "Item 5", "owner_id": "u1" },
    { "id": "a2", "title": "Item 4", "owner_id": "u1" }
  ],
  "count": 5
}
✅ Correct

The first page reports the full dataset size as expected.

2. Second page

GET /api/v1/items/?skip=2&limit=2

200 OK
{
  "data": [
    { "id": "a3", "title": "Item 3", "owner_id": "u1" },
    { "id": "a4", "title": "Item 2", "owner_id": "u1" }
  ],
  "count": 3
}
❌ Wrong

The returned items are plausible, but the total count should still describe all 5 matching items.

3. Third page

GET /api/v1/items/?skip=4&limit=2

200 OK
{
  "data": [
    { "id": "a5", "title": "Item 1", "owner_id": "u1" }
  ],
  "count": 1
}
❌ Wrong

The number of returned rows changes correctly for the last page, but the total keeps shrinking with the offset.

Offset sent Items returned Field value returned Expected field value
0 2 count = 5 5
2 2 count = 3 5
4 1 count = 1 5

## 🗺️ File guide

  1. backend/app/api/routes/items.py — Inspect how the list endpoint builds both the paginated query result and the metadata returned alongside it.
  2. backend/app/models.py — Check what the response schema says the list payload fields are supposed to represent.
  3. backend/tests/api/routes/test_items.py — Review what the current tests assert for list responses and what pagination cases may be missing.

## 💡 Hint

Which response value should describe the total number of matching records independently of pagination, and which values should change when the client asks for a different slice of that same dataset?
