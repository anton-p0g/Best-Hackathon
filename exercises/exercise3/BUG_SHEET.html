Paginated Item Count Drifts With Offset

Section 1 — File changed

backend/app/api/routes/items.py

Section 2 — Code before the change

# backend/app/api/routes/items.py, approximate lines 37-38
    items_public = [ItemPublic.model_validate(item) for item in items]
    return ItemsPublic(data=items_public, count=count)

Section 3 — Code after the change (buggy version)

# backend/app/api/routes/items.py, approximate lines 37-38
    items_public = [ItemPublic.model_validate(item) for item in items]
    return ItemsPublic(data=items_public, count=count - skip)

Section 4 — Why this bug is subtle

Row label Content
Crash? No. The endpoint still returns a valid JSON response and the application keeps running.
First affected path GET /api/v1/items/?skip=1 or any non-zero skip value when listing items.
Unaffected paths Single item lookups, item creation, update, deletion, and item listing with the default skip=0 all appear normal.
Why it survives code review The subtraction looks plausible in pagination code because reviewers often think in terms of remaining results rather than total results.
Logical inconsistency The response field count should describe the full number of matching records, but it is made to depend on the request offset.
User-visible symptom Pagination metadata shrinks as the client moves forward through pages, even though the total matching dataset has not changed.
Why tests may miss it Existing list tests only verify that data is returned and do not assert the count field under paginated requests.
