Password Change Validates the Wrong Password

Section 1 — File changed

target_repo/backend/app/api/routes/users.py

Section 2 — Code before the change

# target_repo/backend/app/api/routes/users.py, approximate lines 84-95
@router.patch("/me/password", response_model=Message)
def update_password_me(
    *, session: SessionDep, body: UpdatePassword, current_user: CurrentUser
) -> Any:
    """
    Update own password.
    """
    verified, _ = verify_password(body.current_password, current_user.hashed_password)
    if not verified:
        raise HTTPException(status_code=400, detail="Incorrect password")
    if body.current_password == body.new_password:
        raise HTTPException(
            status_code=400, detail="New password cannot be the same as the current one"
        )

Section 4 — Why this bug is subtle

Row label Content
Crash? No. The endpoint still returns normal HTTP responses and never throws an unexpected exception.
First affected path PATCH /api/v1/users/me/password when an authenticated user tries to change their password.
Unaffected paths Login, signup, token validation, password recovery, reset-password, and all non-password user profile actions continue to behave normally.
Specific trigger The bug appears only when comparing the submitted passwords during a password-change request; it does not affect requests that do not use both current_password and new_password.
Wrong behavior Users are told their current password is incorrect unless the new password happens to equal the existing password hash check target, which in practice means normal password changes fail silently with a misleading validation error.
Logical inconsistency The code validates the credential that proves identity using the replacement value instead of the current value, mixing two related fields from the same request.
Why it survives code review The function still reads naturally, the variable names are very similar, the verification call signature is unchanged, and the returned error message still looks plausible for failed auth input.
Why tests may miss it If onboarding or smoke tests focus on login and reset-password flows rather than in-session password updates, the regression can slip through.
