Password Change Rejects Valid Credentials

Backend debugging exercise focused on authenticated account management

## 📋 Statement

QA reported that the in-session password change flow is rejecting valid requests for logged-in users. When a user enters their actual current password and a different new password, the API responds with 400 Incorrect password. The odd part is that login itself still works, and password recovery plus reset-password also appear to work normally.

We can reproduce this only on the authenticated self-service password update endpoint. Profile reads, profile edits, and token-based authentication continue to behave as expected. The failure pattern gets stranger when you vary the request: the response seems to depend on the replacement password value more than on the password the user claims is their current one, which makes the error message look misleading.

## 🔴 Observed defective output

1) Login with the existing password

POST /api/v1/login/access-token
Content-Type: application/x-www-form-urlencoded

username=alice@example.com&password=OldPass123!
HTTP/1.1 200 OK
{
  "access_token": "eyJ...",
  "token_type": "bearer"
}
✅ Correct

The account can authenticate successfully with the known current password.

2) Try to change password using the correct current password and a different new password

PATCH /api/v1/users/me/password
Authorization: Bearer eyJ...
Content-Type: application/json

{
  "current_password": "OldPass123!",
  "new_password": "NewPass456!"
}
HTTP/1.1 400 Bad Request
{
  "detail": "Incorrect password"
}
❌ Wrong

The request should succeed because the current password is valid, but it is rejected.

3) Try again with another different replacement password

PATCH /api/v1/users/me/password
Authorization: Bearer eyJ...
Content-Type: application/json

{
  "current_password": "OldPass123!",
  "new_password": "AnotherNew789!"
}
HTTP/1.1 400 Bad Request
{
  "detail": "Incorrect password"
}
❌ Wrong

Changing only the replacement value still produces the same rejection, even though the current password remains valid.

4) Submit the same value for current and new password

PATCH /api/v1/users/me/password
Authorization: Bearer eyJ...
Content-Type: application/json

{
  "current_password": "OldPass123!",
  "new_password": "OldPass123!"
}
HTTP/1.1 400 Bad Request
{
  "detail": "New password cannot be the same as the current one"
}
✅ Correct

This validation still behaves normally, which helps narrow the issue to a more specific branch of the password update logic.

Current password sent New password sent API detail returned Expected result
OldPass123! NewPass456! Incorrect password Password updated successfully
OldPass123! AnotherNew789! Incorrect password Password updated successfully
OldPass123! OldPass123! New password cannot be the same as the current one New password cannot be the same as the current one
