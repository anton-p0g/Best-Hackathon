Debugging Exercise - Item List Count Changes with Pagination

Backend-focused onboarding exercise

## 📋 Statement

QA reported that the item list endpoint returns inconsistent pagination metadata for regular users. When a normal user opens their items page, the first request looks fine, but as soon as the frontend requests the next page using a non-zero skip, the count field in the response starts getting smaller. The actual API call still succeeds and the list data itself looks plausible, so nothing obviously breaks.

We were able to reproduce it only on the paginated list path for non-superusers. Default usage with skip=0 appears correct, and admin users do not seem affected. The frontend team also noticed that the wrong value scales with the pagination offset, which causes the total shown in the UI to keep drifting downward as the user moves through later pages.

## 🔴 Observed defective output

1) First page for a normal user

GET /api/v1/items/?skip=0&limit=2
Authorization: Bearer <normal-user-token>

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a1", "title": "Item 5", "owner_id": "u1", "created_at": "2026-04-12T10:00:00Z" },
    { "id": "a2", "title": "Item 4", "owner_id": "u1", "created_at": "2026-04-12T09:00:00Z" }
  ],
  "count": 5
}
✅ Correct

The first page looks normal: 2 items returned, total count reported as 5.

2) Second page for the same normal user

GET /api/v1/items/?skip=2&limit=2
Authorization: Bearer <normal-user-token>

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a3", "title": "Item 3", "owner_id": "u1", "created_at": "2026-04-12T08:00:00Z" },
    { "id": "a4", "title": "Item 2", "owner_id": "u1", "created_at": "2026-04-12T07:00:00Z" }
  ],
  "count": 3
}
❌ Wrong

The second page still returns valid items, but the total count should remain 5 instead of dropping to 3.

3) Third page for the same normal user

GET /api/v1/items/?skip=4&limit=2
Authorization: Bearer <normal-user-token>

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a5", "title": "Item 1", "owner_id": "u1", "created_at": "2026-04-12T06:00:00Z" }
  ],
  "count": 1
}
❌ Wrong

The number of remaining items on the page is smaller, but the total matching count should still be 5.

Offset sent Items returned Field value returned Expected field value
0 2 count = 5 count = 5
2 2 count = 3 count = 5
4 1 count = 1 count = 5

## 🗺️ File guide

  1. target_repo/backend/app/api/routes/items.py — Inspect how the paginated item list response is assembled and compare the list query with the metadata returned alongside it.
  2. target_repo/backend/app/models.py — Check what the item list response schema promises about the meaning of the count field.
  3. target_repo/backend/tests/api/routes/test_items.py — Review what behavior is currently covered for item listing and what pagination cases are not being asserted.
