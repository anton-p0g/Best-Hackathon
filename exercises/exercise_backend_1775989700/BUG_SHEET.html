Section 4 — Why this bug is subtle

Row label Content
Crash? No. The endpoint still returns a valid JSON response with the expected schema.
First affected path GET /api/v1/items/?skip=>0 for a non-superuser account.
Unaffected paths Single-item reads, item creation, updates, deletes, and the superuser item list path all appear normal.
Visible symptom The count field shrinks as skip increases, even though it should represent the total number of matching items.
Why it survives code review The modified line looks like a plausible pagination adjustment, especially because the list query itself still uses offset(skip) correctly.
Logical inconsistency The response mixes two concepts: total matching rows and the request offset. Only the returned page should depend on skip; the total count should not.
Why tests may miss it The existing item list test only checks that at least two items are returned and does not assert the count field or paginated behavior for normal users.
How the defect compounds Each larger skip value subtracts more from count, making pagination metadata increasingly wrong across later pages.
