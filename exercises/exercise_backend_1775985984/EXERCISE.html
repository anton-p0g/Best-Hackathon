Debugging Exercise

Backend onboarding: investigate an incorrect paginated response

## 📋 Statement

QA reported that the items list endpoint looks fine on the first page, but the total count shown to the UI changes as soon as pagination is used. The actual item rows returned for each page seem reasonable, yet the metadata used for pagination controls starts shrinking when the frontend sends a non-zero skip.

The issue is reproducible on GET /api/v1/items/ when using pagination parameters such as skip=1, skip=2, and higher. Other item endpoints are unaffected, and the default list request without an offset appears normal. QA also noticed that the wrong value scales with the offset, which makes the pagination widget progressively less accurate on later pages.

## 🔴 Observed defective output

1) Default list request

GET /api/v1/items/?limit=2

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a1", "title": "Item A", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T09:00:00Z" },
    { "id": "a2", "title": "Item B", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T08:00:00Z" }
  ],
  "count": 5
}
✅ Correct

The first page returns two items and reports the full matching total.

2) Second page request

GET /api/v1/items/?skip=1&limit=2

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a2", "title": "Item B", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T08:00:00Z" },
    { "id": "a3", "title": "Item C", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T07:00:00Z" }
  ],
  "count": 4
}
❌ Wrong

The returned rows fit the requested offset, but the reported total should still be 5.

3) Later page request

GET /api/v1/items/?skip=3&limit=2

HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": "a4", "title": "Item D", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T06:00:00Z" },
    { "id": "a5", "title": "Item E", "description": "...", "owner_id": "u1", "created_at": "2026-04-12T05:00:00Z" }
  ],
  "count": 2
}
❌ Wrong

The list data remains plausible, but the total keeps decreasing in step with the offset.

## 🗺️ File guide

  1. backend/app/api/routes/items.py — Review how the list endpoint builds its response payload and compare the query inputs with the metadata it returns.
  2. backend/app/models.py — Check what the paginated response schema says the count field is supposed to represent.
  3. backend/tests/api/routes/test_items.py — Look at current coverage for item listing behavior and notice which pagination-specific expectation is missing.
  4. backend/app/crud.py — Confirm whether shared data-access helpers are involved or whether the inconsistency is introduced closer to the route layer.

## 💡 Hint

Which response value should describe the total matching dataset regardless of page position, and which request value should affect only which slice of that dataset is returned?
