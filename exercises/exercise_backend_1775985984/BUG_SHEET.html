Section 4 — Why this bug is subtle

Row label Content
Crash? No. The endpoint returns a valid JSON response and the application keeps running normally.
First affected path GET /api/v1/items/?skip=1 and any other paginated list request where skip > 0.
Unaffected paths GET /api/v1/items/ with the default skip=0, single-item lookup, create, update, and delete endpoints all appear normal.
Why it survives code review The change is a tiny arithmetic adjustment that looks plausible in pagination code, especially if a reviewer mentally conflates total matching records with remaining records after offset.
Logical inconsistency The response field count should describe the total number of matching records, but it is made to depend on the request offset skip.
Observed symptom The list of returned items is correct for the requested page, but the reported total shrinks as pagination advances.
Why beginners can debug it The bug is isolated to one response-building function and can be found by comparing the count query semantics with the response payload contract.
