Debugging Exercise

Users Pagination Metadata Mismatch

## 📋 Statement

QA reported that the users listing endpoint is returning the correct user records for each page, but the pagination information in the response stops making sense once they move away from the default view. On page 1 things look normal, but when they request later pages while keeping the same page size, the metadata appears to change as if the page size were different from what was actually requested.

This only shows up on GET /users when using explicit pagination parameters such as page=2 or page=3 with a fixed items_per_page. Single-user lookups and other user endpoints seem unaffected, and even the list endpoint can look fine at a glance because the number of user objects returned in data still matches the requested limit.

The weird part is that the defect compounds as the requested page number increases. The farther QA paginates, the more the pagination fields drift away from what the frontend expects, even though the actual rows returned still look like the right slice from the dataset.

## 🔴 Observed defective output

1) First page with default-like values

GET /users?page=1&items_per_page=10

200 OK
{
  "data": [{"id": 1}, {"id": 2}, {"id": 3}, {"id": 4}, {"id": 5}, {"id": 6}, {"id": 7}, {"id": 8}, {"id": 9}, {"id": 10}],

    "pages": 35
  }
}
❌ Data length is 10, but the reported page size is 1.

2) Second page with the same requested page size

200 OK

❌ The returned records still follow a page size of 10, but the metadata now claims the page size is 2.

3) Third page with the same requested page size

  1. src/app/api/v1/users.py — Check how the users list endpoint turns database results into the final paginated HTTP response.
  2. src/app/crud/crud_users.py — Confirm what inputs the CRUD layer accepts and whether the actual row selection logic is likely responsible for the mismatch.
  3. src/app/schemas/user.py — Review the user response model so you can separate item serialization concerns from pagination metadata concerns.
