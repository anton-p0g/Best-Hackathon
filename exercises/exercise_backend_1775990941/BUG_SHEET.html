Section 4 — Why this bug is subtle

Row label Content
Crash? No. The endpoint still returns a valid HTTP response and the user list data is still fetched correctly.
First affected path GET /users when page and items_per_page differ, especially on later pages.
Unaffected paths GET /user/{username}, create/update/delete user endpoints, and GET /users calls where page=items_per_page or the client ignores pagination metadata.
Why it survives code review The function call still looks semantically correct because both variables are in scope, both are integers, and both are pagination-related names.
Wrong behavior The response metadata is calculated using the requested page number as if it were the page size, so pagination fields drift while the returned records themselves remain correct.
Specific inconsistency The query uses the real items_per_page to fetch rows, but the response metadata uses page, causing a mismatch between the actual dataset slice and the reported pagination values.
Why beginners may chase the wrong layer Because the database call is correct and the data payload contains the expected users, the bug looks like a frontend pagination issue or a library quirk before it looks like a backend argument mix-up.
Compounding pattern The reported page-size-related metadata changes in proportion to the requested page number, so the defect becomes more obvious only across multiple paginated calls.
