🔧 Debugging Exercise — Conduit Backend Onboarding

📋 Problem Description

Bug report — submitted by a frontend engineer from the web team:

"I am building the paginated article list view and something is wrong with the article count returned by the API. When I load the first page (offset=0) everything looks fine — if there are 23 articles matching a given author filter, the response says articlesCount: 20 (first page, 20 items) and that makes sense. But the moment I fetch the second page (offset=20) to get the remaining 3 articles, the response says articlesCount: 23 even though the articles array only contains 3 items. My pagination logic interprets that as 'there are still more pages to fetch' and sends a third request — which comes back completely empty, with articles: [] and articlesCount: 20."

"I thought it was a frontend bug at first, but I reproduced it with curl against the staging API. It only happens when I paginate past the first page. A simple, unpaginated call (offset=0, default) always gives a consistent count. Single article lookups, the feed endpoint, creating or updating articles — none of those are affected."

"To make things weirder: the larger the offset I use, the larger the fake count gets, even when the article list itself is short. It feels like the count is somehow 'accumulating' the offset value instead of just reflecting how many articles came back."

🔴 Observed Faulty Output

Request sequence that reproduces the problem

Step 1 — First page (looks correct)

GET /api/articles?author=johndoe&limit=20&offset=0
Authorization: Token <valid_token>
{
  "articles": [
    { "slug": "article-1", "title": "Article 1", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-2", "title": "Article 2", "author": { "username": "johndoe" }, "...": "..." },
    "... 18 more items ..."
  ],
  "articlesCount": 20
}
✅ Correct

articlesCount (20) matches articles.length (20). Looks good.

Step 2 — Second page (bug appears)

GET /api/articles?author=johndoe&limit=20&offset=20
Authorization: Token <valid_token>
{
  "articles": [
    { "slug": "article-21", "title": "Article 21", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-22", "title": "Article 22", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-23", "title": "Article 23", "author": { "username": "johndoe" }, "...": "..." }
  ],
  "articlesCount": 23
}
❌ Incorrect

articlesCount is 23, but articles.length is only 3. A client trusting articlesCount will believe a third page exists.

Step 3 — Ghost third page (compounds the damage)

GET /api/articles?author=johndoe&limit=20&offset=40
Authorization: Token <valid_token>
{
  "articles": [],
  "articlesCount": 40
}
❌ Incorrect

articles is empty, but articlesCount is 40 — it keeps growing with the offset value. An infinite scroll client would enter an infinite loop.

Relationship between offset and the inflated count

Request offset sent articles returned articlesCount returned Expected articlesCount
Page 1 0 20 20 20 ✅
Page 2 20 3 23 3 ❌
Page 3 (ghost) 40 0 40 0 ❌

Notice the pattern: articlesCount = articles.length + offset.
The offset value is leaking into the count field.

🗺️ File Guide

Work through these files in order:

  1. app/models/schemas/articles.py — Start here to understand what ListOfArticlesInResponse looks like and what contract articles_count is supposed to fulfill according to the schema.
  2. app/api/routes/articles/articles_resource.py — Find the route handler that builds and returns a ListOfArticlesInResponse; pay close attention to exactly how each field of that response object is populated.
  3. app/api/dependencies/articles.py — Inspect how ArticlesFilters is constructed from incoming query parameters; this will tell you what values are available to the route handler at the time of response construction.
  4. app/db/repositories/articles.py — Verify what filter_articles actually returns and whether the repository itself is aware of the pagination state, to confirm if the issue is in the persistence layer or above it.

💡 Hint

If articlesCount should reflect the number of articles in the current response, why would a value describing the request (instead of the result) need to appear in the expression calculating it?
