🔧 Ejercicio de Depuración — Incorporación Conduit Backend

📋 Descripción del Problema

Reporte de error — enviado por un ingeniero frontend del equipo web:

"Estoy construyendo la vista de lista de artículos paginada y algo está mal con el conteo de artículos que devuelve la API. Cuando cargo la primera página (offset=0) todo se ve bien — si hay 23 artículos que coinciden con un filtro de autor dado, la respuesta dice articlesCount: 20 (primera página, 20 elementos) y eso tiene sentido. Pero en el momento en que obtengo la segunda página (offset=20) para obtener los 3 artículos restantes, la respuesta dice articlesCount: 23 aunque el array articles solo contiene 3 elementos. Mi lógica de paginación interpreta eso como 'todavía hay más páginas para obtener' y envía una tercera solicitud — que regresa completamente vacía, con articles: [] y articlesCount: 20."

"Pensé que era un error de frontend al principio, pero lo reproduje con curl contra la API de staging. Solo sucede cuando pagino más allá de la primera página. Una llamada simple, sin paginar (offset=0, predeterminado) siempre da un conteo consistente. Las búsquedas de artículos individuales, el endpoint de feed, crear o actualizar artículos — ninguno de esos está afectado."

"Para hacer las cosas más extrañas: cuanto mayor es el offset que uso, mayor se vuelve el conteo falso, incluso cuando la lista de artículos en sí es corta. Se siente como si el conteo estuviera de alguna manera 'acumulando' el valor del offset en lugar de simplemente reflejar cuántos artículos regresaron."

🔴 Salida Defectuosa Observada

Secuencia de solicitudes que reproduce el problema

Paso 1 — Primera página (se ve correcto)

GET /api/articles?author=johndoe&limit=20&offset=0
Authorization: Token <valid_token>
{
  "articles": [
    { "slug": "article-1", "title": "Article 1", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-2", "title": "Article 2", "author": { "username": "johndoe" }, "...": "..." },
    "... 18 elementos más ..."
  ],
  "articlesCount": 20
}
✅ Correcto

articlesCount (20) coincide con articles.length (20). Se ve bien.

Paso 2 — Segunda página (aparece el error)

GET /api/articles?author=johndoe&limit=20&offset=20
Authorization: Token <valid_token>
{
  "articles": [
    { "slug": "article-21", "title": "Article 21", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-22", "title": "Article 22", "author": { "username": "johndoe" }, "...": "..." },
    { "slug": "article-23", "title": "Article 23", "author": { "username": "johndoe" }, "...": "..." }
  ],
  "articlesCount": 23
}
❌ Incorrecto

articlesCount es 23, pero articles.length es solo 3. Un cliente que confía en articlesCount creerá que existe una tercera página.

Paso 3 — Tercera página fantasma (agrava el daño)

GET /api/articles?author=johndoe&limit=20&offset=40
Authorization: Token <valid_token>
{
  "articles": [],
  "articlesCount": 40
}
❌ Incorrecto

articles está vacío, pero articlesCount es 40 — sigue creciendo con el valor del offset. Un cliente de scroll infinito entraría en un bucle infinito.

Relación entre offset y el conteo inflado

Solicitud offset enviado articles devueltos articlesCount devuelto articlesCount esperado
Página 1 0 20 20 20 ✅
Página 2 20 3 23 3 ❌
Página 3 (fantasma) 40 0 40 0 ❌

Observa el patrón: articlesCount = articles.length + offset.
El valor del offset se está filtrando en el campo count.

🗺️ Guía de Archivos

Trabaja con estos archivos en orden:

  1. app/models/schemas/articles.py — Comienza aquí para entender cómo se ve ListOfArticlesInResponse y qué contrato se supone que debe cumplir articles_count según el esquema.
  2. app/api/routes/articles/articles_resource.py — Encuentra el manejador de ruta que construye y devuelve un ListOfArticlesInResponse; presta mucha atención a exactamente cómo se puebla cada campo de ese objeto de respuesta.
  3. app/api/dependencies/articles.py — Inspecciona cómo se construye ArticlesFilters a partir de los parámetros de consulta entrantes; esto te dirá qué valores están disponibles para el manejador de ruta en el momento de la construcción de la respuesta.
  4. app/db/repositories/articles.py — Verifica qué devuelve realmente filter_articles y si el repositorio en sí es consciente del estado de paginación, para confirmar si el problema está en la capa de persistencia o por encima de ella.

💡 Pista

Si articlesCount debe reflejar el número de artículos en la respuesta actual, ¿por qué un valor que describe la solicitud (en lugar del resultado) necesitaría aparecer en la expresión que lo calcula?
