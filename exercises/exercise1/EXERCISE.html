Developer Onboarding — Debugging Exercise

📋 Problem Description

Hi, I think there is a bug in the items list endpoint. When I load the first page of items (skip=0) the pagination controls in the UI look completely fine — the total count matches what I see in the database. But the moment I move to the next page (skip=10, skip=20, etc.) the total count returned by the API starts to decrease. By the time I am on the third or fourth page, the frontend thinks there are many fewer items than there actually are and stops rendering the "next page" button altogether.

The strange thing is that the data array itself is always correct — the correct items come back in the correct order. It is only the count field in the response envelope that is wrong. Single item lookups (GET /items/{id}), creations, updates, and deletions behave normally, so it seems to be isolated to the pagination logic of the list endpoint.

One more thing worth noting: the discrepancy is not random or off-by-one — it scales predictably with the skip value I send. The further down the list I paginate, the more incorrect the number gets, which makes me think some request parameter is leaking into a calculation it shouldn't be involved in.

🔴 Observed Faulty Output

Assume the authenticated user owns 42 items in total.

Request 1 — first page (default offset)

GET /api/v1/items/?skip=0&limit=10
Authorization: Bearer <token>
{
  "data": [ /* 10 items */ ],
  "count": 42
}
✅ Correct

count accurately reflects the total number of items owned by the user.

Request 2 — second page

GET /api/v1/items/?skip=10&limit=10
Authorization: Bearer <token>
{
  "data": [ /* 10 items */ ],
  "count": 20
}
❌ Incorrect

count should still be 42 (the total hasn't changed), but it returns 20.

Request 3 — fourth page

GET /api/v1/items/?skip=30&limit=10
Authorization: Bearer <token>
{
  "data": [ /* 10 items */ ],
  "count": 40
}
❌ Incorrect

count should be 42 but returns 40. The frontend now calculates only 4 total pages and hides the "next" button, even though 2 more items exist.

Request 4 — last partial page

GET /api/v1/items/?skip=40&limit=10
Authorization: Bearer <token>
{
  "data": [ /* 2 items */ ],
  "count": 42
}
✅ Coincidentally correct on the last page

skip + len(data) = 40 + 2 = 42 happens to equal the actual total, masking the error at the boundary.

Pattern Summary

skip sent Items returned count returned Expected count
0 10 42 42 ✅
10 10 20 42 ❌
20 10 30 42 ❌
30 10 40 42 ❌
40 2 42 42 ✅ (coincidence)

🗺️ File Guide

  1. backend/app/models.py — Look at the ItemsPublic schema to understand exactly what count is supposed to represent and what type it expects.
  2. backend/app/api/routes/items.py — This is where the list endpoint lives; trace how the value that ends up in the response's count field is produced, and where it comes from.
  3. backend/app/crud.py — Look for any item listing helpers to confirm if the bug is in the query layer or the upper route layer.
  4. backend/tests/api/routes/test_items.py — Check what pagination scenarios the existing tests cover; you might find a gap that allowed this defect to slip through unnoticed.

💡 Hint

The count field is intended to describe a property of the dataset in the database — so why would a value that describes the caller's position within that dataset be part of the calculation?
